# Searchable Artifacts

Lexical full-text search over indexed artifact content, scoped per-project. Powered by SQLite FTS5 in the ProjectDO.

## Overview

tila indexes artifact text content into FTS5 when the artifact's kind is marked `searchable = true` in `tila.schema.toml`. Indexing happens synchronously on the artifact write path — when a `tila artifact put` completes, the content is already searchable. Search results are ranked by BM25 relevance and include text snippets showing matching terms.

## 1. Configuring searchable kinds

Declare a kind as searchable in `tila.schema.toml`:

```toml
[artifacts.lesson]
mime_types = ["text/markdown"]
retention_days = 0
searchable = true
search_mode = "full_text"
```

- `searchable` — Boolean, defaults to `false` when omitted. Set to `true` to include this kind in the FTS5 index.
- `search_mode` — Enum: `"none"` or `"full_text"`. Defaults to `"none"`. Must be `"full_text"` for indexing to occur.

### Built-in searchable kinds

The default `tila.schema.toml` generated by `tila init` ships with these kinds already marked searchable:

| Kind | MIME type | search_mode |
|------|-----------|-------------|
| `lesson` | `text/markdown` | `full_text` |
| `adr` | `text/markdown` | `full_text` |
| `plan` | `text/markdown` | `full_text` |
| `design` | `text/markdown` | `full_text` |
| `review` | `text/markdown` | `full_text` |
| `research` | `text/markdown` | `full_text` |

These are defaults. You can add, remove, or modify searchable kinds by editing `tila.schema.toml` and running `tila schema apply`.

## 2. Supported MIME types

Only text-native MIME types are indexed in v0.1:

- `text/markdown`
- `text/plain`

Artifacts with other MIME types (e.g., `application/pdf`, `image/png`) are stored in R2 but not indexed into FTS5. See [v0.1 limitations](#6-v01-limitations) for details.

## 3. Produced vs. source artifacts in search

tila distinguishes two artifact flavors:

- **Produced artifacts** are emitted by a work unit during work. They have a non-null `resource` field (the work unit ID that produced them).
- **Source artifacts** are uploaded as project-level inputs. They have `resource = NULL`.

Both flavors are indexed when their kind is `searchable = true`. In search results:

- Produced artifacts include a `resource` field identifying the producing work unit.
- Source artifacts have `resource` absent or null.

The `--resource` filter in `tila artifact search` matches produced artifacts only. Source artifacts (resource=NULL) are always included when the kind matches, regardless of the `--resource` filter.

## 4. CLI: tila artifact search

### Tutorial walkthrough

**Step 1:** Declare a searchable kind in `tila.schema.toml` (see [section 1](#1-configuring-searchable-kinds)).

**Step 2:** Apply the schema:

```
tila schema apply
```

**Step 3:** Write an artifact:

```
tila artifact put ./my-lesson.md --task=T-123 --fence=1 --kind=lesson
```

**Step 4:** Search for content:

```
tila artifact search "token rotation"
```

Output (one line per result):

```
produced/T-123/abc123.md  lesson  T-123  Token Rotation Strategy  ...discusses token rotation for fence-based coordination...
```

Each result shows: R2 key, kind, resource (or `(source)` for source artifacts), title (if present), and a text snippet with matching terms.

**Step 5:** Filter by kind:

```
tila artifact search "token rotation" --kind=lesson
```

**Step 6:** JSON output for scripting:

```
tila artifact search "token rotation" --json
```

### FTS5 query syntax

Queries support SQLite FTS5 syntax:

- `AND` — both terms required: `token AND rotation`
- `OR` — either term: `token OR rotation`
- `NOT` — exclude term: `token NOT expired`
- Phrase matching: `"exact phrase"`
- Prefix matching: `token*`

Invalid FTS5 syntax returns a `400 invalid-query` error with a descriptive message.

## 5. UI: Artifacts search view

The Artifacts view in the tila UI includes a search bar at the top. To search:

1. Enter a non-empty query in the search bar
2. Click Search or press Enter
3. Results replace the artifact list, showing kind badges, resource, title, and text snippets
4. Clear the query and submit to return to the artifact list

Search state is ephemeral — it is not URL-persistent. Navigating away from the Artifacts view clears the search.

The kind filter input works in both list mode and search mode — enter a kind name to narrow results.

## 6. v0.1 limitations

The following are explicitly out of scope for v0.1:

- **PDF / OCR:** Artifacts with `application/pdf` or other binary MIME types are stored in R2 but not indexed into FTS5. Text extraction from PDFs is not available.
- **Semantic search:** FTS5 is lexical only — it matches exact terms and prefixes. Embedding-based or semantic similarity search is not available.
- **Global search:** Search is scoped to a single project. Cross-project search is not available.
- **URL-persistent search state:** The UI search bar state is ephemeral. Search queries are not reflected in the URL hash.
